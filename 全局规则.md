# 全局硬性禁令

> ⚠️ 违反任意一条 → **立即停止本轮**，记录原因到执行报告，不论是否达到数量上限。

## 1. 命令串行执行 & 强制随机间隔（最高优先级）

### 🔴 绝对禁令

```
❌ 严禁并发执行任何 CLI 命令
❌ 严禁使用固定间隔（必须每次重新随机）
❌ 严禁跳过 sleep（即使"感觉"不需要）
❌ 严禁在 sleep 期间执行其他命令
```

### ✅ 强制要求

```
✅ 所有命令必须逐条串行执行
✅ 每条命令完成后必须等待 40-55 秒随机间隔
✅ 每次等待前必须重新生成随机数
✅ 宁可慢不可快 — 如果拿不准，取更长的间隔
```

### 📋 执行模板（必须使用）

```bash
# ═══ 单条命令执行模板 ═══
execute_with_delay() {
  local cmd="$1"
  local delay=$((40 + RANDOM % 16))  # 40-55 秒随机值
  
  echo "▶ 执行: $cmd"
  eval "$cmd"
  
  if [ $? -eq 0 ]; then
    echo "✓ 成功，等待 ${delay} 秒..."
    sleep "$delay"
  else
    echo "✗ 失败，跳过等待"
  fi
}

# ═══ 批量执行模板 ═══
commands=(
  'node cli.js search "AI Agent" --count 20'
  'node cli.js get <note_id> --all --depth 1'
  'node cli.js post <note_id> "评论内容"'
)

for cmd in "${commands[@]}"; do
  execute_with_delay "$cmd"
done
```

### 💡 原因

小红书对高频操作敏感，并发或过快间隔会触发风控（限流 / 验证码 / 封号）。这是**硬性下限**，宁可慢不可快。

## 2. 不重复回复

### 🔴 绝对禁令

```
❌ 同一条评论 cid 一生只能被回复一次（自有笔记 + 他人笔记均如此）
❌ 同一作者短期内不重复（推广引流场景）：≥ 7 天才能再次评论同一作者
```

### ✅ 强制要求

```
✅ 执行前必须先调 `node cli.js events --note <note_id>`，扫所有 `post` 操作中带 `--reply-to <cid>` 的记录
✅ 把这些 cid 加入"已回复集合" REPLIED_CIDS，循环里逐条 `if cid in REPLIED_CIDS: skip`
✅ 这是累计禁令，不是单轮去重 — 跨日 / 跨轮均生效
```

### 📋 执行流程

```bash
# ═══ Step 1: 初始化 REPLIED_CIDS ═══
# 从 SQLite comments 表获取所有已回复的 cid
node cli.js replied > /tmp/replied_cids.txt

# ═══ Step 2: 检查是否已回复 ═══
is_replied() {
  local cid="$1"
  grep -q "$cid" /tmp/replied_cids.txt 2>/dev/null
  return $?
}

# ═══ Step 3: 回复前检查 ═══
if is_replied "target_cid"; then
  echo "⏭ 跳过：已回复过该评论"
else
  execute_with_delay 'node cli.js post <note_id> "内容" --reply-to target_cid'
  echo "target_cid" >> /tmp/replied_cids.txt  # 记录已回复
fi
```

## 4. 评论筛选过滤规则

> 💡 评论区数量庞大但有价值的少，盲目回复浪费时间且效果差。

### 核心原则

只回复**有真实互动价值**的评论，跳过无实质内容的。

### 跳过类型（遇到直接跳过）

| 类型 | 典型示例 | 跳过原因 |
|------|----------|----------|
| 祈愿/接龙 | "接好运"、"接offer"、"沾沾喜气" | 纯仪式感，无需回复 |
| 简短感叹 | "真的吗"、"啊？"、"天呐" | 无讨论空间 |
| 求购/求渠道 | "有渠道吗"、"求带"、"怎么买" | 意图不明确，非真实讨论 |
| 祝福/客套 | "恭喜"、"太棒了"、"加油" | 礼貌性评论，无需回应 |
| 疑问句太短 | "这是真的吗？" | 无法判断意图 |

### 优先回复类型

| 类型 | 典型示例 | 回复策略 |
|------|----------|----------|
| 笔记作者评论 | 作者自己的补充说明 | 优先回复，作者通常会看 |
| 有深度的提问 | "这个方案的性能瓶颈在哪？" | 提供有价值的回答 |
| 讨论性评论 | "我觉得这个观点有问题，因为..." | 参与讨论，展示专业度 |
| 分享经验 | "我之前用过类似的，效果是..." | 补充或延伸讨论 |

### 判断标准

```
✓ 评论有实质内容（不是纯感叹/祈愿）
✓ 回复后对方可能会继续讨论
✓ 话题与自身定位相关（AI/技术/编程）
✓ 回复能体现专业度或提供价值
```

## 5. 获取评论限制

### 🔴 绝对禁令

```
❌ 严禁使用 --all 获取所有评论（容易触发风控）
❌ 严禁一次性获取大量评论
```

### ✅ 强制要求

```
✅ 必须使用 --page 1 只获取第一页（通常 20 条）
✅ 如果需要更多评论，可以手动增加 --page 2、--page 3，但要谨慎使用
✅ 每次获取评论后必须等待随机间隔（使用 execute_with_delay 模板）
```

### 📋 执行模板

```bash
# ═══ 获取评论（必须使用 --page 1 限制）═══
execute_with_delay 'node cli.js get <note_id> --page 1 --depth 1'
```

### 💡 原因

- `--all` 会获取所有评论，请求量大，容易触发小红书风控
- `--page 1` 只获取第一页（通常 20 条），足够筛选有价值的评论
- 如果需要更多评论，可以手动增加 `--page 2`、`--page 3`，但要谨慎使用

## 8. 错误处理

### 命令执行失败

```bash
# 在 execute_with_delay 函数中已处理
if [ $? -eq 0 ]; then
  echo "✓ 成功，等待 ${delay} 秒..."
  sleep "$delay"
else
  echo "✗ 失败，跳过等待"
  # 记录失败原因，继续执行下一条命令
fi
```

### 风控触发（错误码 461/471/-10000）

```
1. 立即停止本轮执行
2. 记录触发风控的命令和错误码
3. 等待 30 分钟后再尝试
4. 在执行报告中详细记录
```

### 网络超时

```
1. 不要立即重试（可能是风控）
2. 等待 60 秒后重试一次
3. 如果仍然失败，停止本轮
4. 记录到执行报告
```

### 获取评论触发风控

```
1. 立即停止本轮执行
2. 记录触发风控的命令（是否使用了 --all？）
3. 等待 30 分钟后再尝试
4. 下次执行时必须使用 --page 1 限制
5. 在执行报告中详细记录
```
