# 改进总结

## 问题诊断

### 原始问题
模型在使用 xiaohongshu-cli skill 时，经常忽略网络请求间隔（40-55 秒），导致：
- 触发小红书风控（限流、验证码、封号）
- 命令并发执行
- 重复回复同一评论
- 忽略作者冷却期

### 根本原因
1. **规则是描述性的，不是可执行的** - 模型需要自己构造 sleep 逻辑
2. **关键规则分散在多个文件中** - 模型容易遗漏
3. **缺乏强制性的代码模板** - 模型会"忘记"间隔
4. **没有执行检查点** - 模型不知道当前执行到哪一步
5. **状态追踪不清晰** - REPLIED_CIDS 的维护完全依赖模型手动执行

## 改进方案

### 1. 创建执行模板文件（核心改进）

**文件**：`执行模板.md`

**内容**：
- `execute_with_delay()` - 执行单条命令并等待随机间隔
- `is_replied()` - 检查是否已回复
- `check_author_cooldown()` - 检查作者冷却期（≥ 7 天）
- `init_replied_cids()` - 初始化已回复集合
- 必须使用的执行流程（启动前初始化、执行单条命令、执行回复命令、批量执行）
- 完整的使用示例（互动引流、推广引流）
- 常见错误示例

**效果**：
- 模型可以直接复制使用，不需要自己构造 sleep 逻辑
- 所有关键函数都在一个文件中，不会遗漏
- 强制性的代码模板，模型无法忽略

### 2. 重构 SKILL.md

**改进点**：
- 在"核心安全规则"中添加强制执行模板（带 sleep 的代码块）

- 在"启动前检查"中添加"加载执行模板.md"
- 在"命令映射"中添加获取评论限制（--page 1）
- 在"执行检查清单"中添加间隔执行情况检查和获取评论限制检查
- 使用视觉化的表格和代码块，提高可读性

### 3. 重构全局规则.md

**改进点**：

- 添加强制执行模板（带 sleep 的代码块）
- 添加 REPLIED_CIDS 初始化和检查流程
- 添加执行中检查点（每 5 条命令检查一次）
- 添加错误处理流程（命令失败、风控触发、网络超时）

### 4. 重构互动引流.md

### 6. 重构用户配置.md

**改进点**：

- 添加执行检查清单
- 强调必须使用 `execute_with_delay` 模板

## 改进效果

### 1. 模型无法忽略间隔
- 所有命令必须使用 `execute_with_delay` 模板
- 模板中包含随机间隔（40-55 秒），模型无法跳过
- 使用视觉化的表格和代码块，提高可读性

### 2. 状态追踪自动化
- `init_replied_cids()` 函数自动初始化已回复集合
- `is_replied()` 函数自动检查是否已回复
- `check_author_cooldown()` 函数自动检查作者冷却期

### 3. 规则集中化
- 所有关键函数都在"执行模板.md"中
- 所有规则都在"全局规则.md"中
- 模型只需要加载这几个文件，不会遗漏

### 4. 执行检查点
- 每执行 5 条命令后必须检查执行检查清单
- 如果发现违规，立即停止本轮
- 在执行报告中记录间隔执行情况

### 5. 错误处理完善
- 命令失败时跳过等待，继续执行下一条
- 风控触发时立即停止本轮，等待 30 分钟
- 网络超时时等待 60 秒后重试一次

## 使用方法

### 模型使用流程

1. **启动前初始化**
   ```bash
   # 加载执行模板.md
   # 执行启动前初始化流程
   bash ~/.claude/skills/xiaohongshu-cli/scripts/bridge.sh ensure
   node cli.js status
   init_replied_cids
   ```

2. **执行命令**
   ```bash
   # 所有命令必须使用 execute_with_delay 模板
   execute_with_delay 'node cli.js search "AI Agent" --count 20'
   execute_with_delay 'node cli.js get <note_id> --all --depth 1'
   ```

3. **执行回复**
   ```bash
   # 必须先检查 is_replied 和 check_author_cooldown
   if is_replied "target_cid"; then
     echo "⏭ 跳过：已回复过该评论"
   elif ! check_author_cooldown "target_user_id"; then
     echo "⏭ 跳过：作者冷却期未过"
   else
     execute_with_delay 'node cli.js post <note_id> "内容" --reply-to target_cid'
     echo "target_cid" >> /tmp/replied_cids.txt
   fi
   ```

4. **执行检查**
   ```
   # 每执行 5 条命令后检查
   □ 是否所有命令都是串行执行的？
   □ 每条命令之间是否等待了 40-55 秒？
   □ 是否检查了 REPLIED_CIDS？
   □ 是否检查了作者冷却期（≥ 7 天）？
   □ 是否使用了 ≥ 3 种不同风格？
   □ 是否违反了内容禁令？
   ```

## 总结

本次改进的核心是**把"描述性规则"变成"可执行指令"**：

1. **创建执行模板文件** - 提供必须使用的函数和流程
2. **重构所有 Markdown 文件** - 添加强制执行模板和检查点
3. **集中化规则** - 所有关键函数和规则都在少数几个文件中
4. **自动化状态追踪** - 使用函数自动初始化和检查状态
5. **添加执行检查点** - 每 5 条命令检查一次执行情况

这样，模型在使用 skill 时，**无法忽略网络请求间隔**，因为：
- 所有命令必须使用 `execute_with_delay` 模板
- 模板中包含随机间隔（40-55 秒），模型无法跳过
- 使用视觉化的表格和代码块，提高可读性
- 每 5 条命令必须检查执行检查清单
